# 2026-05-14 公网方言输出故障处理记录

## 当前目标

本轮目标是先保证公网演示稳定可用，同时保留项目关键的音色克隆能力：

- 必须输出目标方言文本对应的音频。
- 未上传参考音频时，输出系统音色方言音频。
- 同时上传主音频和参考音频时，启用 CosyVoice voice enrollment + HTTP TTS 的音色克隆链路。
- 暂时不把 WebSocket 实时流作为公网主输出，改为稳定的 HTTP 音频文件输出。

## 发现的问题

### 1. 音频输入没有进入方言改写

`dialect_service/app.py` 的 `/api/v1/dialect/realtime-session` 旧逻辑只在纯文本输入时调用 `_rewrite_realtime_text_if_enabled()`：

```python
if file is None and text.strip():
    speech_text = rewrite(...)
```

当用户上传主音频时，流程会变成：

```text
音频 -> ASR -> clean_realtime_speech_text -> CosyVoice
```

这会导致普通话 ASR 文本直接进入 TTS，最终输出普通话或普通话味很重的音频。

### 2. 前端 fallback 禁用了方言改写

旧公网前端的 `runFallbackPipeline()` 明确设置：

```javascript
form.set("enable_rewrite", "false");
```

所以实时流失败后，非实时兜底音频也会跳过方言改写，继续输出普通中文。

### 3. `pipeline` 的音频 CosyVoice 分支也绕过了改写

`DialectPipelineEngine.process_audio()` 中，`voice_conversion_provider == "cosyvoice"` 的低延迟分支直接把 ASR 文本清洗后送入 `_build_cosyvoice_tts()`，没有等待 `rewrite_text()`。

这意味着即使前端改成调用 `/api/v1/dialect/pipeline`，上传音频时也仍可能不是方言文本。

### 4. WebSocket 实时流在手机端不稳定

线上日志出现多条：

```text
GET /api/v1/dialect/stream/<id> 404 Not Found
```

说明手机端拿到实时 session 后，后续流连接存在失效或连接方式不稳定的问题。为了比赛演示，当前不再依赖该实时链路作为主输出。

## 本轮修改

### 后端

1. `/api/v1/dialect/realtime-session`
   - 音频输入和文本输入统一调用 `_rewrite_realtime_text_if_enabled()`。
   - 修复“上传音频后直接输出普通话文本”的问题。

2. `/api/v1/dialect/pipeline`
   - 只有 `voice_clone_enabled=true` 时才处理上传参考音频。
   - 只有 `voice_clone_enabled=true` 时才使用默认参考音色。
   - `voice_clone_enabled=false` 时不再偷偷走默认 ref voice。

3. `DialectPipelineEngine.process_audio()`
   - CosyVoice 音频分支现在会先调用 `rewrite_text()`。
   - TTS 使用 `prosody_text -> pronunciation_text -> dialect_text -> semantic_text -> speech_text` 中优先可用的文本。
   - 若 LLM 改写失败，才降级为清洗后的 ASR 文本，并在响应里保留错误信息。

### 前端

公网 `public_web` 改为稳定输出模式：

```text
主音频/文本 -> /api/v1/dialect/pipeline -> 方言改写 -> CosyVoice HTTP 音频文件
主音频 + 参考音频 -> /api/v1/dialect/pipeline -> 方言改写 -> CosyVoice voice enrollment -> 克隆音色方言音频文件
```

当前前端不再：

- 调用 `/api/v1/dialect/realtime-session` 作为主输出。
- 打开 WebSocket。
- 未上传参考音频时使用默认参考音色做“伪克隆”。

当前前端会：

- 设置 `enable_rewrite=true`。
- 设置 `enable_tts=true`。
- 未上传参考音频时设置 `voice_clone_enabled=false`。
- 上传参考音频时设置 `voice_clone_enabled=true` 并上传 `speaker_ref_audio`。
- 展示 `rewrite` 返回的方言文本和最终音频。

## 仍需关注

1. CosyVoice voice enrollment 依赖公网可访问的参考音频 URL 和 DashScope API 稳定性。第一次创建 voice_id 时若外部服务抖动，可能需要重试；缓存命中后速度和稳定性更好。
2. 如果后续重新启用 WebSocket 实时流，需要把它作为增强链路单独验证，不应覆盖当前稳定 HTTP 主链路。
3. 建议后续在页面上明确区分“系统音色方言输出”和“参考音频克隆方言输出”，避免用户误判没有上传参考音频时也应该克隆。
