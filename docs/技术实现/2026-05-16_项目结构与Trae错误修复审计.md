## 项目结构判断

- `FireRedASR2S/asr_service`：ASR 服务入口，负责上传音频归一化、云端 ASR 优先、本地 FireRed ASR fallback。
- `FireRedASR2S/dialect_service`：当前核心编排层，`pipeline_engine.py` 决定 ASR、review、rewrite、TTS、音色复刻与响应结构。
- `FireRedASR2S/fireredasr2s/dialect_pipeline`：底层语音与文本能力，包括 rewrite、Qwen TTS、CosyVoice、Qwen/CosyVoice voice enrollment。
- `FireRedASR2S/public_web`：公网轻量页面，直接调用 `/api/v1/dialect/pipeline`。
- `OpenVoiceRuntime`：本地 OpenVoice audio-to-audio 音色转换后备路线，不是当前公网轻量主链路。

## 发现的路线冲突

5 月 14 日文档认为：

- 音频输入必须先经过 LLM 方言改写。
- TTS 应消费 `prosody_text -> pronunciation_text -> dialect_text -> semantic_text -> speech_text`。
- 这样可以避免普通话 ASR 文本直接进入 TTS。

5 月 15 日 `.trae/specs` 又反向认为：

- CosyVoice 原生支持方言 instruction。
- TTS 应消费原始清洗文本，不应消费 LLM 改写方言文本。
- LLM 改写只用于页面展示。

当前更适合公网主链路的收敛方向：

- CosyVoice/Qwen 声音复刻链路应优先走“原文语义 + 方言 instruction + cloned voice”。
- LLM rewrite 保留为展示、语义校验和文化卡片，不作为复刻方言音频的唯一发音来源。
- 如果没有 cloned voice，仅靠系统音色和普通话文本不能保证方言输出，应尽量启用参考音频 enrollment。

## 已确认的错误

1. `.trae/specs` 标记“instruction always sent”已完成，但代码中仍保留 `_supports_freeform_instruction()`，导致系统音色请求不会携带方言 instruction。
2. 公网页面只有上传额外参考音频才设置 `voice_clone_enabled=true`。这会让“主音频输入 -> 自动复刻该用户音色”的核心链路失效。
3. 公网页面文案仍把主链路描述成“LLM 方言改写 -> TTS”，和当前 CosyVoice instruction 路线不一致。
4. `pipeline_engine.py` 中非 CosyVoice 分支存在早返回后的不可达 Gold Teacher/Voice Matched 旧逻辑，说明历史重构未完全清理。当前公网默认走 CosyVoice，不阻塞本轮修复，但后续切回 Qwen voice clone 时必须专项处理。

## 本轮修复

- `fireredasr2s/dialect_pipeline/cosyvoice.py`
  - 非实时 HTTP 合成请求现在始终带 `input.instruction`。
  - WebSocket 参数也始终带 `instruction`。
  - 若系统音色因 instruction 返回 HTTP 400，会重试一次不带 instruction 的系统音色 fallback，并在结果中记录 `instruction_fallback`，避免直接无音频。
- `public_web/app.js`
  - 主音频、参考音频或文本输入都会设置 `voice_clone_enabled=true`。
  - 有主音频且没有独立参考音频时，后端会自动使用主音频作为参考音色。
- `public_web/index.html`
  - 文案改为“声音复刻 + 方言指令合成”的实际产品路线。
- `tests/test_cosyvoice_public_web.py`
  - 增加回归测试，锁定 CosyVoice instruction 和公网 voice clone 开关。
